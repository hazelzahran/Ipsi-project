Langkah 3 — Kategorisasi FR & NFR
Klasifikasi setiap System Requirement berdasarkan prinsip rekayasa perangkat lunak
ID System Requirement Kategori Jenis NFR Dasar Klasifikasi
SR-01 Sistem menyediakan API untuk menampilkan daftar produk dengan atribut lengkap (foto, kondisi, ukuran, harga, status). Mendukung pagination dan filter kategori/ukuran/harga. FR — Mendefinisikan perilaku spesifik yang dapat diobservasi dan diuji: sistem melakukan sesuatu (menampilkan data). Ini adalah fungsi inti yang dapat diverifikasi dengan test case.
SR-02 Halaman produk harus memuat dalam waktu ≤ 2 detik pada koneksi 4G standar; gambar produk dioptimasi dan di-lazy load. NFR Performance Mendefinisikan seberapa baik sistem melakukan fungsinya (kualitas performa), bukan apa yang dilakukan. Diukur dengan metrik kuantitatif (waktu respons), bukan dengan fitur.
SR-03 Sistem mengelola state keranjang belanja per akun/sesi, memvalidasi stok real-time saat penambahan item, dan menghitung total harga otomatis. FR — Mendefinisikan perilaku fungsional yang spesifik dan dapat diuji: validasi stok, kalkulasi harga, manajemen state. Semua merupakan logika bisnis yang dapat dideskripsikan dalam use case.
SR-04 Sistem mencegah penambahan item dengan stok 0 ke keranjang; menampilkan notifikasi "Item tidak tersedia" kepada pengguna secara jelas. FR — Ini adalah aturan bisnis yang dapat diuji secara fungsional: jika stok = 0, sistem menolak penambahan. Perilaku spesifik dan deterministik, meskipun juga menyentuh UX.
SR-05 Antarmuka keranjang belanja harus intuitif dan responsif, dapat dioperasikan pada perangkat mobile maupun desktop tanpa kehilangan fungsionalitas. NFR Usability Mendefinisikan kualitas pengalaman pengguna, bukan fungsi spesifik. "Intuitif" dan "responsif" adalah atribut kualitas yang diukur melalui uji pengguna, bukan test case fungsional.
SR-06 Sistem menyediakan alur checkout multi-langkah: validasi keranjang → input alamat → kalkulasi ongkir → pilih pembayaran → konfirmasi & pembuatan order dengan ID unik. FR — Mendefinisikan urutan langkah proses bisnis yang spesifik dan dapat diuji secara menyeluruh. Setiap langkah memiliki input, proses, dan output yang terverifikasi.
SR-07 Sistem mengunci stok produk (stock lock) selama sesi checkout aktif dan membebaskan kunci secara otomatis jika checkout tidak diselesaikan dalam 15 menit. FR — Mekanisme penguncian adalah logika bisnis deterministik dengan aturan jelas (lock saat checkout, release setelah timeout). Dapat diuji dengan skenario konkret.
SR-08 Seluruh transaksi pembayaran harus diproses melalui gateway pembayaran berstandar PCI-DSS; data kartu tidak disimpan di server platform. NFR Security Mendefinisikan atribut keamanan (compliance standar PCI-DSS) yang tidak menggambarkan fungsionalitas baru, melainkan batasan dan kualitas dari cara fungsi pembayaran diimplementasikan.
SR-09 Sistem menyimpan rekaman transaksi persisten per akun pengguna dengan status yang dapat diperbarui (menunggu → diproses → dikirim → selesai → dibatalkan). FR — Mendefinisikan mesin status (state machine) yang memiliki aturan transisi jelas dan dapat diuji. Penyimpanan dan pembaruan status adalah perilaku fungsional sistem.
SR-10 Setiap entri riwayat menyimpan snapshot data produk pada waktu transaksi (nama, harga, kondisi, foto) untuk menjaga konsistensi data historis. NFR Reliability Snapshot adalah mekanisme untuk menjamin integritas dan keandalan data historis terhadap perubahan masa depan. Ini adalah atribut kualitas data (reliability/consistency), bukan fitur yang diminta pengguna secara eksplisit.
SR-11 Sistem hanya mengizinkan pengguna berperan Admin mengakses endpoint manajemen produk (RBAC — Role-Based Access Control). NFR Security RBAC adalah kontrol keamanan (access control) yang mendefinisikan siapa yang boleh melakukan apa. Meski terimplementasi secara teknis, ini adalah atribut keamanan sistem, bukan fitur bisnis itu sendiri.
SR-12 Sistem menyediakan form admin untuk menambah produk baru (nama, deskripsi, kondisi, ukuran, harga, kategori, foto) dengan validasi input. FR — Mendefinisikan fungsi CRUD spesifik dengan atribut data yang jelas. Form input dengan validasi adalah perilaku sistem yang dapat diuji secara langsung.
SR-13 Sistem mendukung pengunggahan dan penyimpanan gambar produk ke media storage, dengan validasi format file (JPG/PNG) dan batas ukuran file (maks. 5MB). FR — Upload gambar dengan validasi format dan ukuran adalah perilaku fungsional yang dapat diuji secara langsung dengan file valid dan tidak valid sebagai test case.
SR-14 Operasi penghapusan produk bersifat soft-delete: produk tidak dihapus dari database melainkan diberi flag "dihapus", tersembunyi dari katalog publik namun tetap dapat direferensikan oleh riwayat transaksi. NFR Data Integrity Soft-delete adalah strategi arsitektur data untuk menjaga integritas referensial (referential integrity). Ini adalah keputusan implementasi yang mempengaruhi kualitas dan konsistensi data jangka panjang, bukan fitur yang diminta pengguna.
SR-15 Platform harus tersedia (uptime) minimal 99.5% setiap bulan, dengan mekanisme failover dasar untuk mencegah downtime total pada jam operasional puncak. NFR Reliability Uptime dan ketersediaan sistem adalah atribut kualitas operasional yang diukur secara kuantitatif (persentase SLA), bukan fitur fungsional. Menggambarkan seberapa andal sistem menjalankan fungsinya.
SR-16 Kode sumber harus mengikuti standar penulisan (coding standard) dan didokumentasikan dengan baik untuk memudahkan pemeliharaan dan pengembangan fitur masa depan. NFR Maintainability Maintainability adalah atribut kualitas perangkat lunak yang berkaitan dengan kemudahan modifikasi sistem di masa depan. Tidak dapat diuji sebagai fitur, tetapi diukur melalui code review, kompleksitas siklomatik, dan metrik teknis lainnya.
Thought Process — Prinsip Klasifikasi FR vs NFR:

Pembedaan mendasar terletak pada pertanyaan: "Apakah requirement ini menggambarkan APA yang dilakukan sistem, atau SEBERAPA BAIK sistem melakukannya?"

Functional Requirement (FR) bersifat behavioral — dapat dideskripsikan dalam bentuk use case, memiliki input dan output yang dapat diverifikasi, serta dapat diuji dengan skenario pass/fail yang konkret. Contoh: "Sistem memvalidasi stok sebelum penambahan ke keranjang" — ini dapat diuji langsung.

Non-Functional Requirement (NFR) bersifat qualitative — mendefinisikan atribut kualitas sistem seperti kecepatan (Performance), keamanan (Security), keandalan (Reliability), kemudahan penggunaan (Usability), dan kemudahan pemeliharaan (Maintainability). NFR sering kali tidak terlihat oleh pengguna saat sistem berjalan normal, tetapi sangat terasa dampaknya ketika dilanggar. Dalam konteks rekayasa perangkat lunak, NFR dikenal juga sebagai quality attributes atau -ilities (reliability, scalability, maintainability).
